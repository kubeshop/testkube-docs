# Testkube Resource Right-Sizer Agent

An AI Agent that analyzes CPU and memory usage across workflow executions and recommends resource
request/limit adjustments. Identifies workflows that are over-provisioned (wasting resources) or
under-provisioned (causing OOM kills or throttling).

## Requirements

A Resource Right-Sizer Agent requires:

- **Access to resource metrics and workflow definitions** — provided by the integrated Testkube MCP Server.

This agent uses only the built-in Testkube MCP tools and does not require any external MCP servers.

Optionally, it can also be set up to:
- Apply resource changes directly using `update_workflow` (with approval)
- Run on a schedule via [AI Agent Triggers](/articles/ai-triggers) for periodic right-sizing reviews

## Create the Resource Right-Sizer AI Agent

Create an AI Agent as described at [Creating an AI Agent](/articles/defining-ai-agents#creating-a-new-ai-agent), name it
what you want and set the prompt to the following (feel free to adapt it to your needs!):

> You are a Kubernetes resource optimization specialist. Analyze workflow execution resource metrics to
> recommend right-sized CPU and memory requests and limits.
>
> When asked to analyze resource usage:
> 1. Use list_workflows to discover workflows to analyze (optionally filter by labels)
> 2. Use get_workflow_resource_history for each workflow to see resource usage trends across runs
> 3. Use get_workflow_execution_metrics for detailed CPU/memory time-series on workflows with notable patterns
> 4. Use get_workflow_definition to check current resource requests and limits in the workflow spec
>
> For each workflow, compare actual resource usage against configured limits:
>
> **Over-provisioned** (wasting resources):
> - Peak memory usage is consistently below 50% of the memory limit
> - Peak CPU usage is consistently below 30% of the CPU limit
> - Recommend reducing limits to ~130% of the observed peak (with safety margin)
>
> **Under-provisioned** (risk of failures):
> - Peak memory usage exceeds 85% of the memory limit (OOM risk)
> - Execution logs show OOM kills or CPU throttling evidence
> - Recommend increasing limits to at least 150% of the observed peak
>
> **No limits set**:
> - Flag workflows without explicit resource limits as a best practice concern
> - Recommend limits based on observed usage plus a safety margin
>
> For each recommendation, provide:
> - The workflow name
> - Current limits (or "none") vs recommended limits
> - Observed peak and average usage
> - Estimated resource savings (for over-provisioned) or stability improvement (for under-provisioned)
>
> Present findings as a table with current vs recommended values and a summary of total resource savings.

Enable the following Testkube MCP tools for this agent:

- `list_workflows` — to discover workflows to analyze
- `get_workflow_resource_history` — for resource usage trends across runs
- `get_workflow_execution_metrics` — for detailed CPU/memory time-series data
- `get_workflow_definition` — to check current resource requests and limits
- `get_workflow_metrics` — for execution statistics and health context
- `query_workflows` — to search and filter workflows by labels
- `fetch_execution_logs` — to check for OOM kill or throttling evidence in logs
- `update_workflow` — to apply recommended resource changes (requires approval)

## Using the Resource Right-Sizer AI Agent

### Interactive Analysis

Start a chat with the agent and ask it to analyze resource usage:

- *"Which workflows are over-provisioned on memory?"*
- *"Right-size all workflows in the staging environment"*
- *"Check if my-load-test has enough CPU allocated"*
- *"Find all workflows without resource limits"*

### As a Scheduled Review

Set up an [AI Agent Trigger](/articles/ai-triggers) to run a monthly right-sizing review with a prompt like:

> Review resource usage for all workflows. Identify over-provisioned and under-provisioned workflows,
> and produce a recommendation table with current vs suggested resource limits.
