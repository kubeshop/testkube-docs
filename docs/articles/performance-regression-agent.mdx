# Testkube Performance Regression Detector Agent

An AI Agent in Testkube can be set up to detect performance regressions in your test workflows by comparing
recent execution times and resource usage against historical baselines.

## Requirements

A Performance Regression Detector Agent requires:

- **Access to workflow metrics and execution history** — provided by the integrated Testkube MCP Server.

This agent uses only the built-in Testkube MCP tools and does not require any external MCP servers.

Optionally, it can also be set up to:
- Monitor specific workflows by label (e.g. `type=performance`, `suite=load-tests`)
- Run on a schedule via [AI Agent Triggers](/articles/ai-triggers) for proactive daily or weekly regression checks

## Create the Performance Regression Detector AI Agent

Create an AI Agent as described at [Creating an AI Agent](/articles/defining-ai-agents#creating-a-new-ai-agent), name it
what you want and set the prompt to the following (feel free to adapt it to your needs!):

> You are an AI agent that detects performance regressions in test workflows. Analyze execution duration trends
> and resource usage patterns to find workflows that are getting slower or consuming more resources over time.
>
> When asked to check for regressions:
> 1. Use list_workflows to discover workflows (optionally filter by labels like type=performance)
> 2. For each workflow, use get_workflow_metrics to get execution statistics and health scores
> 3. Use get_workflow_resource_history to analyze resource trends (CPU, memory) across runs
> 4. Use get_workflow_execution_metrics for detailed time-series resource data on suspicious workflows
> 5. Use query_executions to compare recent execution durations against older runs
>
> Flag workflows where:
> - Recent execution durations are significantly higher than the historical average (e.g. >20% increase)
> - Resource usage (CPU or memory) shows an upward trend across recent runs
> - Health scores are declining even if tests are still passing
>
> For each flagged workflow, provide: the workflow name, the metric that regressed, the magnitude of change,
> and when the regression started. Use get_workflow_definition to inspect the workflow for possible causes
> (e.g. new steps, changed images, different parallelization).
>
> Present findings as a summary table and suggest concrete next steps for each regression.

Enable the following Testkube MCP tools for this agent:

- `list_workflows` — to discover workflows to analyze
- `get_workflow_metrics` — for execution statistics and health scores
- `get_workflow_resource_history` — for resource trend analysis across runs
- `get_workflow_execution_metrics` — for detailed CPU/memory time-series data
- `query_executions` — to compare durations across many executions
- `list_executions` — to find recent runs for specific workflows
- `get_workflow_definition` — to inspect workflow config when diagnosing causes

## Using the Performance Regression Detector AI Agent

You can use this agent in several ways:

### Interactive Analysis

Start a chat with the agent and ask it to check for regressions:

- *"Check all my performance test workflows for regressions"*
- *"Has the duration of my-load-test increased over the last week?"*
- *"Which workflows are getting slower?"*

### As a Scheduled Trigger

Set up an [AI Agent Trigger](/articles/ai-triggers) that runs this agent on a schedule (e.g. daily) to
proactively detect regressions before they become problems. Configure the trigger with a prompt like:

> Analyze all workflows with the label type=performance for duration and resource regressions over the last 7 days.
> Report any workflows where execution time increased by more than 15% or resource usage shows an upward trend.
